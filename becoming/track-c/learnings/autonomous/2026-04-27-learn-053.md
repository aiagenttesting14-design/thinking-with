# OpenClaw Sub-Agent Coordination Patterns: A Practical Orchestration Playbook for TestBot

**Research Date:** April 27, 2026  
**Task ID:** learn-053  
**Word Count:** ~1,050 words

---

## Executive Summary

OpenClaw’s sub-agent system is powerful, but the community pattern is surprisingly consistent: keep the main agent thin, spawn isolated workers for bounded tasks, and treat handoffs as the real architecture problem. Official docs emphasize isolation, push-based completion, and selective use of `context: "fork"`. Community reports and GitHub issues add the caution: multi-agent setups become expensive, fragile, and hard to debug when the orchestrator delegates too much context, relies on persistent sub-agent identity, or lets workers improvise their own communication patterns.

For TestBot, the practical conclusion is not “use more agents.” It is: use a disciplined hub-and-spoke model with short-lived workers, structured task payloads, explicit acceptance criteria, and minimal transcript inheritance. The goal is to make delegation cheaper, more reliable, and easier to audit.

---

## 1. The Right Default: Orchestrator + Ephemeral Workers

The strongest official guidance is that sub-agents are isolated by default and should stay that way unless there is a specific reason not to. OpenClaw’s docs describe sub-agents as background runs in their own sessions, with separate context and token usage, and recommend `context: "fork"` only when the child genuinely needs the current requester transcript. Community practice lines up with this: orchestrator-only mains and disposable workers are the dominant pattern.

That matters because it prevents the most common failure mode in multi-agent systems: confusing coordination with collaboration. If every agent inherits broad context, all of them become expensive and noisy. If each worker gets only the task, inputs, and finish condition it needs, coordination stays legible.

**Practical rule for TestBot:** default every spawned worker to isolated context and require a written reason before using `fork`.

---

## 2. Handoffs Matter More Than Spawn Mechanics

The naïve assumption is that spawning is the hard part. It isn’t. The hard part is specifying the job clearly enough that the worker can finish without follow-up. GitHub feature requests around `announceTarget`, persistent named sessions, and sub-agent templates all point to the same gap: users want more durable, reusable handoff structure.

The community workaround is consistent:
- pass a structured task block
- include parent goal, scope, constraints, and output shape
- define what “done” means
- require concise return format

For TestBot, every delegation should include five fields:
1. **Objective** — what the worker is trying to achieve
2. **Inputs** — files, URLs, or prior findings it may rely on
3. **Constraints** — length, safety, no-write limits, tool restrictions
4. **Deliverable** — exact output format
5. **Done condition** — the test for success

That is more important than model choice. A mediocre worker with a crisp handoff often beats a stronger worker with vague instructions.

## 3. Use Sub-Agents for Bounded Work, Not Ongoing Roles

Official docs are clear that `/subagents spawn` is one-shot and that persistent work should use `sessions_spawn` with thread/session mode instead. Community feedback adds a sharper framing: sub-agents are for “small tasks only” unless you build around their limits.

The main limitation is identity durability. GitHub issue #19780 highlights that UUID-based sub-agent sessions are not a stable long-term coordination primitive across restarts. In practice, this means sub-agents should not be treated like durable employees with memory. They are better thought of as contractors with a single assignment.

**Good uses for TestBot sub-agents:**
- focused research on one source cluster
- file summarization
- parallel comparison work
- draft generation from a tight spec
- validating one narrow question

**Bad uses:**
- long-lived role ownership
- memory-dependent planning across days
- vague “keep working on this” tasks
- workflows that require repeated back-and-forth clarification

If the role needs continuity, use a persistent session or sibling specialist agent instead.

---

## 4. Keep the Main Agent Out of the Weeds

A good orchestrator should not also be the busiest worker. Several community discussions recommend keeping the main agent as a coordinator only, delegating specialized execution outward and preserving its context window for planning, synthesis, and user interaction.

This is especially relevant for TestBot because it already has broad responsibilities: research, memory upkeep, automation, Slack communication, and system improvement. If the main session also does every deep retrieval or long browser workflow itself, it becomes the bottleneck and accumulates noisy context.

A better split is:
- **Main agent:** decide, sequence, synthesize, communicate
- **Worker agents:** gather, test, summarize, draft
- **Persistent specialists (optional):** hold domain-specific continuity where needed

This also improves failure handling. When a worker fails, the main agent is still clean enough to retry, switch approaches, or escalate.

---

## 5. Design for Push Completion, Not Polling

OpenClaw explicitly warns against polling `subagents list`, `sessions_list`, or `sessions_history` in loops just to wait. Completion is push-based, and cron orchestration prefers final descendant output over stale interim text. That suggests a practical discipline: spawn, continue useful work, and only inspect status when there is a debugging or intervention reason.

For TestBot, this means two habits:
- never build orchestration around busy-wait checks
- separate “waiting” from “monitoring” in the design

If a child is working, the parent should either do other work or yield. Polling wastes tokens and clutters transcripts.

---

## 6. The Cost and Complexity Ceiling Is Real

OpenClaw’s own FAQ says multi-agent setups are often more expensive and less efficient than one bot with separate sessions. Reddit users echo that some avoid sub-agents entirely because of bugs and instead route visible work through sibling agents or Discord-based orchestration. GitHub requests for deterministic spawning, templates, governed orchestration, and nesting controls all imply the same truth: once the team structure grows, manual coordination costs begin to dominate.

So the play is not “build a swarm.” It is “earn complexity slowly.”

**Recommended ceiling for TestBot right now:**
- one orchestrator
- up to 2–3 parallel ephemeral workers per task
- optional persistent specialist only when a domain repeatedly justifies it

Beyond that, debugging cost rises sharply.

---

## 7. Immediate Improvements for The Becoming System

Here are the concrete changes worth adopting now:

### A. Standardize the delegation template
Create one reusable spawn prompt format for all sub-agent tasks with objective, inputs, constraints, deliverable, and done condition.

### B. Add a “why fork?” check
Only use `context: "fork"` when the worker truly needs live requester transcript state. Default to isolated.

### C. Cap parallelism intentionally
Use a soft cap of 2–3 concurrent workers unless the task is pure research or comparison.

### D. Separate specialists from sub-agents
If a role needs continuity, use a persistent session or sibling agent, not a disposable worker.

### E. Log orchestration failures
Track failed handoffs, ambiguous worker outputs, and over-delegation cases in WORKING.md or a dedicated ops note. This becomes architectural feedback, not just incident cleanup.

### F. Preserve the orchestrator’s cleanliness
Keep the main agent focused on synthesis and decision-making. Resist the urge to make it do all execution too.

---

## Bottom Line

The best OpenClaw sub-agent pattern is boring on purpose: one clean orchestrator, isolated workers, structured handoffs, push-based completion, and minimal inherited context. That is not the most cinematic architecture, but it is the one most likely to make TestBot more reliable. The mistake to avoid is treating sub-agents like a magical intelligence multiplier. They are really a coordination system, and coordination quality determines whether they help or hurt.
