# learn-052 — OpenClaw error handling and recovery patterns

**Date:** 2026-04-27  
**Topic:** OpenClaw error handling and recovery patterns — how resilient agents handle tool failures, API errors, and unexpected states

OpenClaw’s failure modes are not random. Across GitHub issues, community threads, and Playwright guidance, the same pattern shows up repeatedly: agents fail when they treat every error as the same class of problem. The practical lesson is that reliability comes less from “retry more” and more from classifying failures correctly, using the smallest safe recovery move first, and switching strategy early when the system is clearly degraded.

The strongest evidence comes from three clusters of failures. First, browser-path failures: issues like #14503 and #66514 describe a system that can look healthy at the status layer while `act`, `snapshot`, or `screenshot` are effectively dead. Second, transcript/session corruption failures: issues like #6682, #7930, and related reports show that rate limits, timeouts, or mid-stream tool errors can leave sessions in broken states where future valid work fails immediately. Third, provider-layer failures: issue #26463 highlights silent model-provider failures where invalid keys, unstable APIs, or empty responses produce no actionable diagnosis.

The practical takeaway for TestBot is to adopt a four-bucket error model:

1. **Transient execution errors** — timeout, 429, temporary 5xx, brief network failure  
2. **Stale state errors** — old refs, dead `targetId`, tab-not-found, session drift  
3. **Structural corruption errors** — orphaned tool results, invalid session history, repeated impossible API errors  
4. **Configuration/auth errors** — invalid API keys, missing browser path, persistent policy/permission failures

Each bucket needs a different response.

## 1. Retries should be narrow, not blind

The community evidence strongly supports limited retries for transient failures only. A single timeout or network wobble should usually get one controlled retry. A second failure of the same step is a signal, not bad luck. In browser issue #45661, one operator’s field-tested policy was simple and good: if the same action fails twice in one browser stack, switch stacks instead of retrying blindly. That principle generalizes beyond browsers.

For TestBot, the default retry policy should be:

- **429 / rate limit / timeout / temporary 5xx:** retry once after a short backoff
- **same failure twice in a row:** stop retrying that exact path
- **third attempt only if something changed:** different tool, fresh snapshot, new target, new session, or new model

This prevents the classic failure spiral where the agent wastes time and tokens hammering a broken path.

## 2. Treat stale state as a refresh problem, not a logic problem

OpenClaw browser failures often come from stale state rather than bad reasoning. Snapshot succeeds, then `act` fails. Restarting a profile may revive the browser, but the old `targetId` can immediately become invalid. Reusing stale references after a mutation is one of the most consistent browser failure modes reported in issues like #44431 and #14503.

So the right recovery ladder is:

- re-snapshot after page mutation or navigation
- keep actions on the same known tab
- if `tab not found`, discard old `targetId` completely
- if `act` fails while `snapshot` still works, treat the interaction path as degraded
- if browser status says healthy but snapshot/screenshot/act disagree, trust behavior over status

This matters because OpenClaw sometimes reports health at one layer while the real control path is dead. A resilient agent should privilege observed tool behavior over optimistic health checks.

## 3. Distinguish recoverable failures from poisoned sessions

The most dangerous failure class is not a timeout. It is a poisoned session that keeps producing nonsense errors. Issue #6682 is especially useful here: after several 429s, a later valid tool call failed almost instantly with a synthetic “missing tool result” error. Timing showed it was not a real missing result; the transcript had become inconsistent.

This gives TestBot an important rule: **when an error appears impossibly fast or semantically mismatched, suspect session corruption.** Examples:

- a tool call “times out” in milliseconds
- a fresh tool call instantly reports missing prior tool output
- repeated API 400 errors reference tool IDs or history states that no longer make sense
- a valid next step fails before execution could realistically begin

Recovery here should not be “try again harder.” It should be:

- stop using the damaged path
- start a fresh session or delegate the task to a clean sub-agent
- summarize the failure mode in plain terms
- avoid replaying a long broken transcript if a fresh child can do the task faster

This is one of the clearest places where sub-agents are not just convenient but operationally safer.

## 4. Prefer graceful degradation over binary failure

A strong agent should rarely go from “working” to “I can do nothing.” It should step down capability. If browser interaction is broken, web fetch or search may still extract the needed information. If one model provider is unstable, another may still work.

A useful degradation ladder for TestBot:

- **Browser full interaction** → browser snapshot/read-only inspection  
- **Browser snapshot** → `web_fetch` extraction  
- **Direct completion in current session** → clean sub-agent session  
- **Primary model/provider** → fallback provider or lighter model  
- **Complex workflow** → narrowed minimal next action

This keeps momentum. It also makes failures legible: the agent can say, in effect, “interactive path is degraded, but I can still complete 70% of the task another way.”

## 5. Logging should preserve diagnosis, not just symptoms

Issue #26463’s complaint about silent failures is broader than provider auth. The real lesson is that agents need structured error notes, not vague recollections. When TestBot hits a meaningful failure, the useful log fields are:

- tool/provider involved
- exact operation attempted
- error class (transient, stale state, corruption, config/auth)
- whether retry was attempted
- whether state was refreshed
- whether fallback path succeeded

This creates reusable memory. Over time, patterns become visible: “browser `act` is fragile after idle,” “429s sometimes poison long sessions,” “this site needs slower typing.”

## 6. Concrete operating playbook for TestBot

The best immediate implementation is a small decision tree:

**If timeout / 429 / 5xx:**  
Retry once with backoff. If repeated, switch path.

**If stale ref / tab-not-found / browser act mismatch:**  
Re-snapshot or reopen tab. Never reuse stale refs or target IDs after restart.

**If error is instant but doesn’t make causal sense:**  
Assume transcript/session corruption. Move work to fresh session or sub-agent.

**If auth/config appears wrong:**  
Stop early, surface the real blocker, do not loop.

**If task can still succeed another way:**  
Degrade gracefully and continue.

## Recommended changes for The Becoming System

1. **Adopt the four-bucket error taxonomy** in working docs and future skills.  
2. **Cap retries at one blind retry**; any third attempt must include a changed condition.  
3. **Use fresh sub-agents as a corruption escape hatch** for suspicious session-history failures.  
4. **Add lightweight failure logging** to autonomous workflows so repeated breakpoints become visible over time.  
5. **Build fallback-first habits**: browser → fetch, current session → child session, primary provider → alternate provider.  
6. **Trust observed tool behavior over nominal health checks** whenever the two disagree.

The core lesson is simple: resilient agents do not try to overpower failure. They identify the failure class, make the smallest valid recovery move, and abandon broken paths early. That is how TestBot gets more reliable without becoming more brittle.
