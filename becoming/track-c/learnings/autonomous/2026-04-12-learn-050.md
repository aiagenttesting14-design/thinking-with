## Executive Summary

After surveying OpenClaw's multi-agent documentation, community configurations, and real-world implementations from power users, the community has converged on five major architecture patterns — each with distinct tradeoffs. The core insight: sub-agents are fully isolated brains with no shared memory, no user context, and no session continuity. Everything must be passed in the task prompt. This constraint shapes every viable multi-agent pattern. For TestBot specifically, the Hub-and-Spoke pattern is the most immediately actionable, but the typed pipeline pattern offers the highest reliability for complex workflows.

## The Five Community Architecture Patterns

### 1. Hub-and-Spoke (Orchestrator → Workers)

The most common and recommended starting point. One orchestrator agent (typically running Claude Opus) receives all requests, breaks them into subtasks, delegates to specialist workers running cheaper/faster models, and synthesizes the results.

```
Human
  |
Orchestrator (Opus)
  /    |    \
Coder Research  Ops
(Docker) (Docker) (Docker)
```

**How it works in OpenClaw:**
- Orchestrator bound to all channels (Slack, Telegram, etc.)
- Delegates via `sessions_spawn` to specialist sub-agents
- Sub-agents run in Docker sandboxes with restricted tools
- Results announce back to orchestrator → human

**Config pattern:**
```json
{
  "agents": {
    "list": [
      { "id": "main", "workspace": "~/.openclaw/workspace", "sandbox": { "mode": "off" } },
      { "id": "coder", "workspace": "~/.openclaw/workspace-coder", "sandbox": { "mode": "all", "scope": "agent" } },
      { "id": "research", "workspace": "~/.openclaw/workspace-research", "sandbox": { "mode": "all" } }
    ]
  },
  "tools": {
    "agentToAgent": { "enabled": true, "allow": ["main", "coder", "research"] }
  }
}
```

**Strengths:** Simple, predictable routing, easy to debug, clear ownership  
**Weaknesses:** Single point of failure at orchestrator; context bottleneck at orchestrator; no parallel orchestration

**Used by:** TheSethRose/OpenClaw-Advanced-Config (Seth → Linus coder → Finch research → Otto cron)

### 2. Dream Team (Multi-orchestrator with 7+ specialists)

The most elaborate community pattern — Adam91holt's "Kev's Dream Team" runs 14+ agents with distinct models per specialty: Opus for orchestration, Codex for code, Flash for research (cheap+fast), Gemini Pro for visual.

```
Human (Adam)
  |
Kev (Orchestrator, Opus 4.5)
  /  |  |  |  \  \
Rex Hawk Scout Dash Dot Pixel ...
```

**Key architectural decisions:**
- Orchestrator (Kev) NEVER writes code — only delegates and synthesizes
- Each specialist has its own identity files (IDENTITY.md, SOUL.md, AGENTS.md)
- Shared coordination files: TEAM_PROTOCOL.md (no-spam rules, deliverable formats), TEAM.md (roster, escalation), GLOSSARY.md (shared language)
- GitHub-centric execution: Issues as source of truth, PRs as deliverables
- Self-healing docs: Agents update their own .md files when they learn something

**Cost insight from the community:** "We don't risk a cheap model on a production database migration script (that's for Codex). Don't burn Opus credits on summarizing search results (that's for Flash)."

**Strengths:** Maximum specialization; model arbitrage by task type  
**Weaknesses:** High complexity; 14+ agents require dedicated maintenance; coordination overhead significant

### 3. GodMode Workflow Templates (Predefined pipelines)

Predefined workflow templates prevent the orchestrator from reinventing the process every time. The system auto-selects the workflow based on task type.

```
Human: "Add user authentication with JWT"
  |
GodMode Orchestrator auto-selects workflow:
  |
NEW FEATURE workflow:
@researcher → @architect → @builder → [@validator + @tester] (parallel) → @scribe
```

**Five predefined workflows:**
- New Feature: researcher → architect → builder → [validator + tester] → scribe
- Bug Fix: builder → [validator + tester]
- API Change: architect → api-guardian → builder → [validator + tester] → scribe
- Research: researcher → report
- Release: scribe → github-manager

**Key insight:** Typed pipelines beat prompt engineering for coordination. A YAML file with condition, loop, and stdin piping is infinitely more reliable than telling an LLM "if the review is negative, go back to step 2, but only up to 3 times."

**Strengths:** High reliability through predefined structure; parallel quality gates  
**Weaknesses:** Inflexible for novel tasks; requires careful workflow design upfront

---

### 4. Blackboard (Shared state with mutexes)

Framework-agnostic — connects 12 agent frameworks (LangChain, AutoGen, CrewAI, OpenAI Assistants, OpenClaw) through shared state with mutexes for atomic coordination.

**What makes it different:** Agents commit atomically to shared state; conflicts are resolved through mutex patterns; enables cross-framework collaboration.

**Community pattern:** Content quality gates — automated hallucination/placeholder detection; budget tracking — track token spend per article with ceiling (e.g., $5/article max), kill runaway sub-agents.

**Strengths:** Framework-agnostic; enables complex parallel coordination  
**Weaknesses:** Requires careful mutex management; complex setup

---

### 5. Pipeline Pattern (Typed workflows, highest reliability)

Typed pipelines beat prompt engineering for coordination. YAML-based condition, loop, and stdin piping is the community consensus for production reliability.

**Best practice from community:**
- Budget tracking: Set ceiling per task (e.g., $5/article max)
- Kill switch: Auto-terminate runaway sub-agents that exceed budget
- Quality gates: Automated hallucination/placeholder detection after each review round
- Self-healing docs: Reviewers update their own checklist with new patterns they caught — compounds quality over time

## Critical Technical Constraints

### Sub-agent Memory Is Empty

This is the most important constraint in OpenClaw multi-agent architecture:

> "Sub-agents have no memory and no user context — the orchestrator must pass everything in the task prompt."

Sub-agents receive only AGENTS.md + TOOLS.md — no SOUL.md, USER.md, IDENTITY.md, or MEMORY.md. This means:
- Every task prompt must be self-contained
- No cross-session continuity for workers
- Orchestrator owns all context synthesis

### Session Isolation

Background agent runs spawned from an existing agent:
- Run in isolated sessions (`agent::subagent:`)
- Announce results back to requester chat automatically
- Non-blocking: returns `runId` immediately
- Default concurrency: 8 (`maxConcurrent`)
- Auto-archive after 60 minutes (configurable)

### Nested Depth Limits

- maxSpawnDepth: 2 (Main → Orchestrator → Worker)
- Depth-1 orchestrator gets: sessions_spawn, subagents, sessions_list, sessions_history
- Depth-2 workers get NO session tools (leaf workers)
- maxChildrenPerAgent: 5 (default), range 1-20
- Cascade stop: killing parent kills all children

### Cross-Agent Communication

- Must be explicitly enabled: `tools.agentToAgent.enabled: true`
- Allowlisted: `tools.agentToAgent.allow: ["agent1", "agent2"]`
- Sends a message into another agent's session with reply-back

## What TestBot Should Adopt

### Immediate (Low Complexity, High Value)

**1. Hub-and-Spoke for Newsletter Production**

The most practical near-term pattern for The Becoming System:
- Orchestrator (TestBot main) coordinates
- Research sub-agent: handles web research for article topics
- Writing sub-agent: handles draft generation
- Main agent handles editing, publishing, Slack reporting

This mirrors how the community's "research team" pattern works — parallel research + synthesis.

**2. Typed Pipeline for Autonomous Learning Sessions**

Current cron job pattern: single agent does research → write → post. Replace with a pipeline:
```
Researcher agent → Editor agent → Poster agent (to Slack)
```

Each stage is a separate isolated agent run. If research fails, pipeline stops without running the writer. This prevents the "write a report about nothing because research silently failed" problem.

**3. Self-Healing Documentation Protocol**

After each autonomous learning session, the sub-agent updates a `learnings/protocol.md` with new patterns it discovered. Over time, this compounds into a living knowledge base that doesn't require the main agent to remember everything.

### Medium-Term (Requires Configuration Changes)

**4. Model Arbitrage by Task Type**

Currently TestBot uses the same model (MiniMax-M2 or Claude Sonnet) for all tasks. The community shows clear model-per-role patterns:
- Orchestration: Claude Opus (complex reasoning)
- Routing: MiniMax-M2 or Flash (fast, cheap)
- Research: Tavily-augmented agent (search-optimized)
- Writing: Sonnet 4 (creative, conversational)

This can reduce token costs by 35-50% while maintaining output quality.

**5. Shared Coordination Files**

Add to workspace:
- `TEAM_PROTOCOL.md`: deliverable formats, quality standards, no-spam rules
- `TEAM.md`: agent roster, escalation paths, current active tasks
- `GLOSSARY.md`: shared terminology to prevent miscommunication between agents

These files solve the "sub-agents have no context" problem by giving workers explicit reference documents.

### Not Recommended (Too Complex for Current Scale)

The Dream Team pattern (14+ agents) is explicitly NOT recommended for TestBot's current setup:
- Coordination overhead exceeds benefit at TestBot's operational scale
- 30+ cron jobs already create management complexity
- Would require dedicated maintenance time that isn't available
- Revisit when: 5+ distinct operational domains, dedicated monitoring, full workspace per agent

## Key Takeaways

1. **Sub-agents are empty brains** — no memory, no context. Every task prompt must be self-contained. This is the #1 architectural constraint.

2. **Typed pipelines outperform prompt engineering** — YAML-based conditionals and loops are more reliable than LLM judgment calls for workflow control.

3. **Hub-and-Spoke is the right starting point** — simple, debuggable, maps to TestBot's actual workflow (orchestrate → delegate → synthesize).

4. **Model arbitrage is real cost savings** — don't burn Opus credits on tasks that Flash or MiniMax-M2 can handle.

5. **Self-healing documentation compounds** — agents that update their own protocol files create institutional knowledge that survives session boundaries.

The most actionable immediate change: split the autonomous learning cron into a typed pipeline with researcher → synthesizer → poster agents, each in isolated sessions with explicit delivery configuration.
