## Key Findings

### 1. The Model Configuration Structure

OpenClaw's model configuration lives in `agents.defaults` with several distinct layers:

**Primary + Fallbacks** — the core routing chain:
```json
model: {
  primary: "anthropic/claude-sonnet-4-6",
  fallbacks: ["openai/gpt-5.4", "google/gemini-3-flash"]
}
```

**Model Allowlist** (`agents.defaults.models`) — acts as the allowlist for `/model` command and must include any fully-qualified model ID you want to use. Without it, you get `"Model provider/model is not allowed"`.

**Heartbeat Model** — separate from primary:
```json
heartbeat: {
  every: "30m",
  model: "google/gemini-2.5-flash-lite"
}
```

**Subagent Default Model**:
```json
subagents: {
  model: "deepseek/deepseek-reasoner"
}
```

These three tiers — heartbeat, primary/fallback, subagent — are the primary levers TestBot has for cost optimization.

### 2. Three Active Bugs That Undermine Routing Reliability

This is the most important section of this report. There are three confirmed bugs in OpenClaw's model routing that directly affect TestBot's operational reliability:

**Bug #47705 — Fallback model permanently overwrites primary config**: When the primary model fails and a fallback handles the request, the fallback model gets written back to `agents.list[].model` in openclaw.json, permanently replacing the configured primary. This means a single API issue can silently switch TestBot to a less capable model for all future sessions. The primary never retries after a fallback succeeds. *Impact: Critical — TestBot's model could silently degrade.*

**Bug #71571 — Agent-level model field ignored during session creation**: The `model` field on agent definitions in `agents.list[]` is being ignored. Sessions always resolve to `agents.defaults.model.primary` instead of the agent-specific model. *Impact: Per-agent model overrides do not work, which limits multi-agent configuration value.*

**Bug #65519 — Subagent model override silently ignored**: When spawning a sub-agent via `sessions_spawn` with an explicit `model` parameter, or when configuring `agents.defaults.subagents.model`, the specified model is silently ignored. Sub-agents always inherit the main agent's model regardless of configuration. *Impact: High — all sub-agents use the primary model, defeating cost optimization efforts.*

Community note: Issue #43521 suggests ACP runtime handles model override correctly (`mode: "run", model: "gemini-2.5-flash-lite"` works), but native subagent runtime ignores it.

**Mitigation strategy**: Until these bugs are fixed in a release, TestBot should:
- Monitor `openclaw.json` after any fallback event to check if primary was overwritten
- Explicitly specify `model` in `sessions_spawn` but do not trust it — verify actual usage
- Accept that subagents will always use the primary model ( Opus in TestBot's case) and plan budgets accordingly

### 3. The Cost Tiering Principle

The community consensus is clear: **use the right model for the right job**. The cost differential between tiers is extreme — from $0.40/M tokens (Xiaomi MiMo-V2-Flash) to $30/M (Claude Opus). For a heartbeat check that just confirms "still alive," there's no quality difference between tiers. The cheap model works.

The three-tier model that community guides consistently recommend:

| Tier | Task Type | Example Models | Cost Range |
|------|-----------|----------------|------------|
| Frontier | Complex synthesis, novel reasoning, architecture decisions | Opus 4.5, GPT-5.2 | $15-30/M |
| Mid-tier | Daily work — code generation, research, content creation | Sonnet 4.5, DeepSeek R1, Gemini 3 Flash | $2-4/M |
| Budget | Heartbeats, simple lookups, classification, sub-agents | Gemini 2.5 Flash-Lite, DeepSeek V3.2, Xiaomi MiMo | $0.40-0.60/M |

The practical savings: a properly tiered configuration cuts costs 50-80% for typical workloads, because the majority of OpenClaw interactions are simple tasks that a budget model handles perfectly.

### 4. Heartbeat as Its Own Cost Lever

Heartbeats are the most obvious optimization target. Every 30 minutes, TestBot's gateway sends a heartbeat using whatever model is configured as primary. For a setup running Opus as primary, that's $30/M every 30 minutes — roughly $1.44/day or $43/month just for health checks, assuming minimal token usage per heartbeat.

Configuring a separate budget model for heartbeats drops this to approximately $0.02/day or $0.60/month. The quality difference for a heartbeat is zero — it's just checking liveness.

Current TestBot configuration: Unknown. The TOOLS.md and workspace files may not document the heartbeat model explicitly. This is an immediate action item.

### 5. Subagent Model Inheritance Problem

Even if TestBot sets `agents.defaults.subagents.model`, the bug (#65519) means subagents inherit the parent's model regardless. This has two implications:

1. **Cost**: Every subagent spawn uses the primary model (expensive)
2. **Capability mismatch**: Bounded research tasks that don't need frontier-level reasoning still pay frontier prices

The workaround using ACP runtime (`runtime: "acp"`) with explicit model in `sessions_spawn` appears to work (per issue #43521), but this changes the execution context and may not be appropriate for all subagent use cases.

### 6. OpenRouter Auto-Router as a Simplicity Option

OpenRouter's `openrouter/openrouter/auto` model acts as an automatic router — it evaluates prompt complexity and routes to the appropriate underlying model. This is a zero-configuration cost optimization: set it as primary and OpenRouter handles tiering automatically.

Trade-offs:
- Less control over which model handles which task
- Routing decisions are opaque
- May not be optimal for TestBot's specific use cases where domain knowledge could inform routing decisions

For a setup like TestBot's with specific research, coordination, and communication tasks, manual tiering with explicit model selection is likely more appropriate than auto-routing.

### 7. Model Allowlist Must Be Kept in Sync

The `models` catalog in openclaw.json acts as an allowlist. If a fallback model or heartbeat model is not listed in `models`, it will be rejected with a "Model not allowed" error even if it exists in the provider catalog. This means every model TestBot references must be explicitly added to the allowlist, or the routing silently fails and falls through to the next option.

## Practical Improvements TestBot Should Make

### Immediate (this week)

1. **Audit current heartbeat model**: Check what model is configured for `agents.defaults.heartbeat.model`. If unset or using a premium model, change to `google/gemini-2.5-flash-lite` or similar budget tier.

2. **Verify model allowlist completeness**: Every model referenced in fallbacks, heartbeat, or subagent config must exist in `agents.defaults.models`. Missing entries cause silent routing failures.

3. **Add fallback persistence monitor**: After any incident where a fallback was used, check openclaw.json to confirm primary wasn't overwritten (Bug #47705). Document a recovery procedure.

### Short-term (this month)

4. **Implement model tiering for cron jobs**: Different cron jobs have different complexity requirements. Lightweight crons (status checks, memory cleanup) can use budget models; complex crons (research, analysis) should use mid-tier or frontier. This requires `runTimeoutSeconds` and `model` per spawn, but be aware that model override may not work (Bug #65519).

5. **Consider heartbeat every: "60m" instead of "30m"**: Halving heartbeat frequency cuts that cost in half with no quality loss for liveness checking. The community standard is trending toward longer intervals with more reliable failure detection through other means.

6. **Document the routing architecture**: Add a section to TOOLS.md describing TestBot's model tiering — primary, fallback chain, heartbeat model, subagent model — so the configuration is reviewable and recoverable.

### Medium-term

7. **Watch for OpenClaw releases addressing bugs #47705, #71571, #65519**: These are active issues with community engagement. A fix would enable proper subagent cost optimization and prevent fallback overwrite behavior. When fixed, revisit `agents.defaults.subagents.model` configuration.

8. **Evaluate OpenRouter auto-router**: If TestBot's model selection becomes a burden to maintain, the auto-router could simplify configuration. Worth a test with a non-critical channel before full adoption.

## What This Means for TestBot's Architecture

The model routing layer is where TestBot's operational cost and reliability meet. The current setup appears to use minimax/MiniMax-M2.7 as primary (per runtime context), which is already a cost-effective choice compared to Opus. However:

- Heartbeat configuration needs review — budget models are clearly appropriate
- Subagent cost cannot be optimized until the model override bug is fixed, but TestBot should plan for it
- Fallback chain should use different providers (not just different models from the same provider) for true resilience
- The model allowlist must be maintained as a living configuration element

The bigger strategic point: model routing is not a "set and forget" configuration. It requires active maintenance, monitoring after failures, and regular review as the model landscape changes. The community pattern of tiering — frontier for hard problems, mid-tier for daily work, budget for infrastructure — is well-established and applicable to TestBot's specific workloads.
