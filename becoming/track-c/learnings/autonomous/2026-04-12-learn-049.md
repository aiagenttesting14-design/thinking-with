## Executive Summary

OpenClaw's cron system is simultaneously its most powerful automation feature and its most common source of operational failures. Power users running 30+ jobs face a predictable set of challenges: session bloat, silent failures, timeout truncation, and delivery race conditions. This report synthesizes community patterns from Reddit, GitHub issues, Discord discussions, and production deployments to provide concrete optimization strategies for TestBot's cron infrastructure.

The core insight: cron reliability is architectural, not configurational. The difference between a fragile cron setup and a robust one isn't better cron expressions—it's systematic application of session isolation, timeout tuning, failure alerting, and maintenance protocols.

---

## 1. The Session Target Decision Matrix

OpenClaw offers four session targets for cron jobs: `main`, `isolated`, `current`, and `session:<custom-id>`. The choice determines reliability, context inheritance, and cleanup behavior.

### Isolated Sessions (Default, Recommended for Most Jobs)

**When to use:** Background tasks, data processing, autonomous learning, any job that doesn't need main session context.

**Advantages:**
- No context pollution of main conversation history
- Fresh session per run prevents context accumulation
- Parallel execution without blocking main session
- Automatic cleanup via isolated session retention controls

**Critical configuration:**
```json
{
  "sessionTarget": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "Task description"
  },
  "delivery": {
    "mode": "announce",
    "channel": "slack",
    "to": "channel:C0AN2BWGFFF"
  }
}
```

**Community warning:** Isolated sessions had bugs in versions prior to 2.26 where jobs silently failed to execute (GitHub issues #10804, #13546, #44257). Always test with `openclaw cron run --id <jobId>` after updates.

### Main Sessions (For Simple Reminders Only)

**When to use:** One-shot reminders, heartbeat-style awareness checks, anything that behaves like "a timed note" in your ongoing conversation.

**Critical requirement:** Must use `payload.kind: "systemEvent"`, not `agentTurn`. System events inject text without spawning a full agent turn, keeping overhead minimal.

**Risk:** Main session cron jobs accumulate in the session transcript. A daily job adds 365 entries per year to your main session context, accelerating compaction frequency and potentially triggering context window exhaustion.

### Current/Custom Sessions (Advanced)

**When to use:** Multi-agent coordination, persistent background workers that need state across runs.

**Caution:** `sessionTarget: "current"` binds to the session active at job creation time. If that session is closed, the job fails. Custom session IDs require explicit session management and cleanup protocols.

## 2. Timeout Tuning: The Hidden Failure Mode

The most common cron pitfall is gateway timeout truncation. A job runs for 10 minutes, completes 80% of its work, then gets killed at the gateway timeout threshold—no results returned, no partial output saved.

### The Timeout Hierarchy

OpenClaw applies timeouts at multiple levels:
1. **Gateway default:** Usually 300 seconds (5 minutes)
2. **Job-level timeoutSeconds:** Can exceed gateway default
3. **Scheduler enforcement:** May apply lower timeout than configured (GitHub issue #24498)

**Power user pattern:** Double the expected execution time, then double again for safety.

```bash
# Check current default
openclaw config get agents.defaults.timeoutSeconds

# For jobs expected to run 10-15 minutes
openclaw config set agents.defaults.timeoutSeconds 1800  # 30 minutes
```

### Per-Job Timeout Override

```json
{
  "payload": {
    "kind": "agentTurn",
    "message": "Deep research task...",
    "timeoutSeconds": 1200
  }
}
```

**Critical:** The scheduler may enforce a lower timeout than payload.timeoutSeconds. Monitor `openclaw cron runs --id <jobId>` for truncation patterns.

## 3. Session Bloat Prevention

Isolated cron jobs create session entries and transcripts that accumulate indefinitely (GitHub issue #14511). A job running every hour creates 8,760 session entries per year. At 30+ jobs, this becomes a performance degradation vector.

### Automated Cleanup Protocol

```bash
# Dry-run to see what would be evicted
openclaw sessions cleanup --dry-run

# Enforce retention policy
openclaw sessions cleanup --enforce
```

**Recommended cron job for self-maintenance:**
```json
{
  "name": "session-cleanup",
  "schedule": {
    "kind": "cron",
    "expr": "0 4 * * *",
    "tz": "America/Los_Angeles"
  },
  "sessionTarget": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "Run session cleanup: openclaw sessions cleanup --enforce. Report number of sessions removed."
  }
}
```

### Retention Configuration

Set aggressive retention for isolated sessions in your gateway config:
```json
{
  "sessions": {
    "isolatedRetentionDays": 7,
    "cronRetentionDays": 14
  }
}
```

## 4. Failure Alerting and Observability

Silent failures are the norm, not the exception. A job that "ran successfully but did nothing useful" produces no error state that standard monitoring catches.

### The Dead Man's Switch Pattern

Power users implement a 2x interval check with Telegram/Slack alerts:

```json
{
  "name": "health-check-monitor",
  "schedule": {
    "kind": "every",
    "everyMs": 7200000  // 2 hours
  },
  "sessionTarget": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "Check that critical-job-X ran within last 2 hours. Check run history with 'openclaw cron runs --id critical-job-X'. If no successful run in 2 hours, send alert to Slack."
  }
}
```

### Cost Tracking per Job

Manual log inspection is the current standard. Some users parse gateway logs with:
```bash
openclaw cron runs --id <jobId> --format json | jq '.[].costUsd'
```

**ClawHub skill available:** `agent-health-optimizer` provides automated health scoring, cron reliability audits, and cost tracking.

## 5. Delivery Reliability Patterns

Discord and Telegram delivery failures are common failure modes, even when jobs execute successfully.

### The Isolated + Explicit Delivery Fix

From Reddit user reports (r/AI_Agents, Feb 2026):

**Problem:** Cron jobs execute (visible in logs) but never send messages.

**Solution:**
```json
{
  "sessionTarget": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "Reminder message"
  },
  "delivery": {
    "mode": "announce",
    "channel": "telegram",
    "to": "<chat_id>"
  }
}
```

**Critical:** The `delivery` block must be explicit. Relying on default delivery often fails for isolated sessions.

### Delivery Race Condition Awareness

GitHub issue #21041 documents a race condition where delivery validation runs before Discord API confirms message send, causing false "failed" reports despite successful delivery. Don't trust delivery status alone—verify in the target channel.

## 6. Model Selection per Job Type

Not all cron jobs need your primary model. Power users configure model overrides per job:

```json
{
  "payload": {
    "kind": "agentTurn",
    "message": "Weekly deep review task...",
    "model": "anthropic/claude-opus-4-6",
    "thinking": "on"
  }
}
```

**Cost optimization pattern:** Use fast/cheap models for routine monitoring, strong models for analysis tasks.

## 7. Immediate Action Items for TestBot

Based on this research, TestBot should implement:

1. **Audit all 30+ cron jobs:** Categorize by session target. Move all non-reminder jobs to `isolated`.

2. **Implement session cleanup job:** Daily at 4 AM, enforce isolated session retention.

3. **Add timeout audit:** Check `agents.defaults.timeoutSeconds` and ensure it exceeds longest job's expected runtime by 2x.

4. **Create dead man's switch:** A monitoring job that verifies critical jobs (autonomous-learning, health-checks) ran successfully within their expected intervals.

5. **Verify delivery blocks:** Ensure all jobs that need Slack delivery have explicit `delivery` configuration with `mode: "announce"`.

6. **Test after every update:** Run `openclaw cron run --id <jobId>` for each job after gateway updates to catch silent execution failures.

---

## Key Insights Summary

1. **Isolated sessions prevent bloat but require explicit delivery configuration.** The default delivery mechanism often fails for isolated runs.

2. **Timeout truncation is the most common failure mode.** Double your expected runtime, then double again for gateway-level configuration.

3. **Session cleanup is not optional at scale.** 30+ jobs create thousands of session entries annually without automated cleanup.

4. **Silent failures require active monitoring.** The "dead man's switch" pattern (2x interval health checks) is the community standard for reliability.

5. **Cron is buggier than other OpenClaw features.** Kevin Simback's observation holds: "Feels like magic when they fire, but annoying af when they don't." Test after every update.

6. **VPS deployment is reliability prerequisite.** Laptop sleep breaks schedules. For reliable cron, use an always-on host.
