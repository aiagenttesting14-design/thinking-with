# OpenClaw Memory Architecture in Practice: How TestBot Should Structure Durable Recall

OpenClaw’s memory system is strongest when treated as architecture, not magic. The official docs make the core point plainly: there is no hidden memory. Durable recall comes from files on disk, search over those files, and optional consolidation layers like dreaming and memory-wiki. Community discussion adds a second practical lesson: most “memory failures” are not retrieval model failures but system design failures—important facts were never written down clearly, the agent was not instructed to search first, or the memory stack became too complicated to trust. For TestBot, the goal should be a simple, inspectable memory architecture that improves continuity without adding fragile infrastructure.

The first practical rule is to separate durable truth from temporary context. OpenClaw already gives the right primitives: `MEMORY.md` for long-term facts and decisions, `memory/YYYY-MM-DD.md` for daily notes, and optional `DREAMS.md` for reviewable consolidation output. That structure should stay the center of gravity. TestBot should treat `MEMORY.md` as a compact operating manual: durable user preferences, standing rules, core project facts, critical safety boundaries, and stable workflow policies. Daily notes should hold transient observations, progress notes, and unresolved questions. The mistake many users describe is letting `MEMORY.md` become a giant journal. Once that happens, injection cost rises, signal drops, and the model starts ignoring the very file meant to preserve continuity.

A good working pattern is a four-tier memory stack. Tier 1 is bootstrap memory: `SOUL.md`, `USER.md`, `AGENTS.md`, `TOOLS.md`, and a lean `MEMORY.md`. These are always-available identity and operating files. Tier 2 is topic memory: small focused files for recurring domains such as Substack publishing, interview outreach, browser workflows, or Track C operational lessons. Tier 3 is daily logs under `memory/YYYY-MM-DD.md`. Tier 4 is cold archive: older reports, transcripts, and research artifacts that are not injected automatically but remain searchable. The advantage is mechanical clarity. When TestBot needs a stable rule, it belongs in Tier 1 or Tier 2. When it needs historical context, it should search Tiers 3 and 4 instead of bloating the bootstrap set.

The second rule is “search before synthesis.” Community threads repeatedly point out that agents often guess from fuzzy recollection instead of checking memory tools. That is preventable. TestBot already has a strong instruction to run `memory_search` before answering questions about prior work, decisions, dates, people, preferences, or todos. The operational improvement is to extend that habit beyond explicit memory questions. Before any task that depends on prior protocol—Substack posting, browser automation, outreach rules, cron maintenance, or queue updates—TestBot should search memory for the relevant domain first. This is especially important because external reports and session summaries can become stale or partial. Retrieval should be a gate, not an optional enhancement.

The third rule is to keep memory searchable by structure, not just content. OpenClaw’s default memory search combines semantic similarity with keyword matching, and QMD goes further with reranking and query expansion. That means file naming, headings, IDs, and explicit terminology matter. TestBot should write notes with stable anchors: task IDs like `learn-051`, exact file paths, workflow names like “RSS verification protocol,” and short headings that match how future searches will be phrased. This is more important than prose elegance. Memory retrieval works best when the note contains both conceptual language and exact strings. A practical habit would be to end important reports or protocol docs with a short “Operational anchors” section listing workflow names, IDs, and key terms.

Fourth: prefer simple reliability over ambitious automation. The docs position dreaming and memory-wiki as useful additions, but not replacements for core memory. That matches community reality. Dreaming can help promote recurring patterns into long-term memory, and memory-wiki can turn loose notes into structured claims, dashboards, and digests. But recent issue reports around bridge mode importing zero artifacts show that newer layers can introduce operational uncertainty. For TestBot, the right move is conservative adoption. Keep `memory-core` and file-based memory as the source of truth. Treat dreaming as optional reviewable consolidation, not a dependency. Treat memory-wiki as a future enhancement only after the base file architecture is clean and stable. In other words: do not build critical continuity on top of a layer that still appears operationally noisy.

Fifth: compaction safety matters more than perfect retrieval. The docs highlight automatic memory flush before compaction, and community operators emphasize reserve capacity as a defensive measure. The practical lesson is simple: if important context exists only in chat, it is already at risk. TestBot should assume long sessions will compact and should externalize important decisions quickly. The best time to write memory is immediately after a durable decision, not later during cleanup. For ongoing workflows, that means updating the relevant file as part of the task itself. A browser reliability discovery belongs in the browser protocol notes the same day. A new outreach boundary belongs in the outreach documentation immediately. Durable learning should happen inline with work, not as a separate idealized consolidation pass that may never happen.

What should TestBot change now? Five concrete moves stand out.

1. **Audit `MEMORY.md` for size and role creep.** Keep only durable facts, standing preferences, and high-value rules. Move procedural detail into topic files.
2. **Create topic memory files for recurring operations.** Good first candidates: browser automation, Substack publishing, interview outreach, and Track C operational lessons.
3. **Use memory search as a workflow gate.** Before memory-dependent tasks, search first, then act.
4. **Write with retrieval in mind.** Include exact IDs, filenames, protocol names, and concise headings so hybrid search has both semantic and keyword hooks.
5. **Defer advanced memory layers until the core is clean.** Keep dreaming optional and treat memory-wiki as experimental until reliability stabilizes.

Bottom line: TestBot does not need a more exotic memory stack right now. It needs a stricter one. Clear tiers, small durable files, mandatory retrieval before memory-dependent work, and immediate externalization of important decisions will improve continuity more reliably than adding another plugin. The best memory architecture is the one that survives compaction, survives session resets, and is simple enough to trust under pressure.
