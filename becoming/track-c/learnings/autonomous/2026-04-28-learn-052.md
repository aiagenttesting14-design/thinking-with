# OpenClaw Error Handling and Recovery Patterns: A Practical Playbook for TestBot

OpenClaw is powerful, but the failure mode is rarely a clean crash. More often it is a half-success: a browser action timed out after doing the thing, a cron run exited before delivery, a sub-agent completed but the orchestrator never harvested the result, or memory behaved inconsistently because the right retrieval path was not triggered. The practical lesson from docs, release notes, and GitHub issues is simple: reliability in OpenClaw comes less from one magic setting and more from designing every workflow around bounded failure, explicit verification, and clean recovery.

The first pattern is to separate **execution** from **verification**. In OpenClaw, tool success is not the same as task success. Browser automation is the obvious example, but the same logic applies everywhere. If TestBot publishes something, the browser click is not proof of publication; RSS or a fetch of the live artifact is proof. If a cron job says it ran, that is not proof the downstream delivery happened; a visible output file, session artifact, or posted message is proof. This should become a standard rule: every important workflow needs one lightweight success check that is independent of the tool that performed the action.

Second: put hard boundaries on loops. Community discussion around retry spirals and sub-agent cascades points to the same risk: OpenClaw will often keep trying unless the workflow designer decides where to stop. That means max iterations, explicit retry counts, and clear escalation paths are not optional. For TestBot, the practical version is: retry once for transient UI or network failure, retry twice only when the operation is idempotent, and after that stop and log the block instead of improvising forever. Silent failure is cheaper than infinite failure, but bounded failure is better than both.

Third: timeouts need to be deliberately generous at the tool layer and deliberately strict at the workflow layer. Too-short per-call timeouts create fake failures, especially with browser and model-backed memory behaviors. Too-long whole-job timeouts create hanging cron runs and resource buildup. The best operational split is: tool actions get enough time to complete realistically, while the full workflow gets a firm upper bound. In practice for TestBot, browser and memory-heavy operations should often have 30-second action assumptions, while cron jobs and autonomous sessions should have a higher-level timeout that kills the whole run if recovery does not happen quickly.

Fourth: recover with a fixed playbook, not intuition. OpenClaw’s issues around browser state, active memory, and dreaming-managed cron behavior all point to the same operational truth: once the system is in a weird state, cleverness usually makes it worse. Recovery should be mechanical. For browser work, that means stop, restart, re-snapshot, retry. For memory or dreaming anomalies, that means inspect status, confirm the plugin is actually active, confirm the relevant cron or sweep exists, and only then continue. For sub-agent work, it means check whether the child session actually completed before spawning a replacement. The point is to reduce duplicate work and state confusion.

Fifth: preserve idempotency whenever a task can be retried. This matters most for publishing, logging, queue mutation, and message sending. Before writing a new file, check whether today’s task file already exists. Before appending a completion note, verify the task was not already marked completed. Before posting a Slack summary, make sure the run did not already send one. Many OpenClaw failures are ambiguous—especially when a tool times out after the external system may already have acted. Idempotent workflow design turns ambiguity from danger into inconvenience.

Sixth: treat memory as part of error handling, not separate from it. Several memory-related issues show that recall can fail silently, dreaming sweeps may not run as expected, and active memory only applies in narrow eligible session types. That means reliability cannot assume memory will rescue a workflow later. Critical rules need to live in durable files and in the current task prompt when the work is high-risk. For TestBot specifically, operational protocols like browser reset steps, queue mutation rules, and publication verification standards should stay in workspace files rather than relying on retrieval happening at the right moment.

Seventh: isolate background helpers, but do not over-fragment work. Sub-agents are useful for bounded research or transformation, but every child introduces another possible completion-routing failure. The most reliable pattern is one orchestrator plus a small number of isolated workers with explicit deliverables. If the task is short enough to finish directly, direct execution is often safer than delegation. If delegation is necessary, the handoff should include objective, inputs, constraints, and the exact return artifact. The bigger lesson is that coordination overhead is itself an error surface.

Eighth: log the failure in a reusable way. A one-off recovery that lives only in chat history is wasted pain. The strongest operators turn recurring failures into compact protocols: “if active memory is enabled but not injecting, verify chat type eligibility,” or “if dreaming is enabled but no managed cron appears, inspect doctor/status before assuming config is wrong.” TestBot should keep appending these patterns into durable operational notes so the same category of failure gets cheaper over time.

The most useful immediate upgrades for TestBot are concrete. First, add a verification step to every external action workflow. Second, enforce a default retry policy: one retry for transient failures, then stop and log. Third, raise action-level patience while lowering workflow-level tolerance for hanging runs. Fourth, document fixed reset sequences for browser, sub-agent, and memory failures. Fifth, design every queue, file, and messaging mutation to be idempotent. Sixth, keep critical operational knowledge in files, not only in session memory.

The deeper lesson is that OpenClaw reliability is architectural, not emotional. The fix is not “try harder next run.” The fix is to make success verifiable, failure bounded, recovery mechanical, and repeated mistakes durable as protocol. That is the difference between an impressive demo agent and one Stephen can actually trust.
