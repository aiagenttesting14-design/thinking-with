## How OpenClaw's Context Actually Works

OpenClaw's context window is not memory ‚Äî it's the active token space inside the model's current window. What lives in context:

- System prompt (agent rules, tools, skills, time, runtime)
- Injected workspace files (MEMORY.md, AGENTS.md, SOUL.md, etc.)
- Current session transcript (conversation history)
- Tool call internals (truncated at file size limits)

Context is not the same as memory. Memory lives on disk and can be reloaded. Context evaporates when the window fills or the session resets.

The 200K-token context window sounds large. It isn't. Workspace files (especially this system's multi-file architecture), tool schemas, and session history consume 60-80K tokens before a single substantive conversation message. Long sessions, large files, and verbose tool outputs can trigger compaction within 30-50 messages.

## Compaction: What Happens and Why It Hurts

When context approaches overflow, OpenClaw's compaction system kicks in. It summarizes older conversation history into a compact entry, preserving facts and tool results but destroying the conversational thread ‚Äî tone, rhythm, continuity of thought, implicit context from earlier exchanges.

The symptoms of unmanaged compaction:

- The agent "forgets" earlier parts of the conversation even though facts were technically preserved
- References to earlier decisions become opaque ("as we discussed") without the agent being able to cite specifics
- The agent's sense of narrative continuity frays session by session
- Eventually: `context_length_exceeded` error ‚Äî the session becomes unusable

For TestBot, compaction is particularly damaging because The Becoming's core value proposition is continuous, evolving partnership. A compacted session loses the thread of that partnership.

## The Community's Three-Layer Solution

### Layer 1: Compaction Configuration (Safety Net)

The community standard config for `agents.defaults.compaction` in `openclaw.json`:

```json
"compaction": {
  "mode": "safeguard",
  "reserveTokensFloor": 40000,
  "memoryFlush": {
    "enabled": true,
    "softThresholdTokens": 4000,
    "systemPrompt": "Session nearing compaction. Save all important facts, pending decisions, and working state to appropriate memory files before the window resets."
  }
}
```

**Key insight:** `reserveTokensFloor: 40000` means OpenClaw triggers compaction when 40K tokens remain ‚Äî providing a safety buffer before the window actually overflows. The `memoryFlush` prompt explicitly instructs the agent to write its current state to memory files before summarization destroys it.

**Critical bug (v2026.3.1 regression):** Issue #32106 documents an aggressive compaction loop when `memoryFlush` is enabled. If TestBot enables this, monitor for compact‚Üíflush‚Üícompact‚Üíflush cycles. The workaround is setting `mode: "summarize"` temporarily until the regression is fixed.

**Another critical bug (#17034):** `softThresholdTokens` is a fixed absolute value (default 4000), not relative to context window size. When switching between models with different context sizes (200K ‚Üí 1M tokens), the threshold becomes either too aggressive or too lenient. Manually adjust `softThresholdTokens` when changing models.

**Immediate action:** Run `/context list` in TestBot's current session to see exactly what is consuming tokens. This is the diagnostic command the community considers essential.

### Layer 2: File-Based Memory Architecture (The Real Continuity Layer)

Files are the only durable memory. Chat-based instructions are ephemeral. The community's tiered memory model:

- **Tier 1 (Bootstrap):** MEMORY.md, AGENTS.md, SOUL.md ‚Äî always injected, must be under 4K tokens each
- **Tier 2 (Topic files):** Created on demand for active projects (`becoming/track-c/`, `becoming/interviews/`)
- **Tier 3 (Daily logs):** Session summaries written to `memory/YYYY-MM-DD.md` after each session
- **Tier 4 (Cold archive):** Old daily logs and completed project files

**Pre-compaction backup pattern (from r/ClaudeAI):** Before any manual reset or when the session is healthy, save ~20K tokens of raw conversation (human + assistant messages only, no tool internals) to `memory/conversation-pre-compact-[date].md`. This preserves the conversational thread that compaction destroys.

**The Memory Flush prompt matters:** The default memory flush prompt is generic. Customizing it for TestBot's specific needs ‚Äî "Remember where The Becoming project stands, what Stephen's current priorities are, what tasks are pending" ‚Äî significantly improves what gets preserved.

### Layer 3: Session Type Selection (Architectural Choice)

The three session types serve different continuity needs:

- **Main session:** Default, persistent across messages, shares context with cron-triggered agent turns. Risk: context accumulation from mixed topics.
- **Isolated session:** Fresh sessionId per run, no context inheritance. Right for: stateless background tasks, one-shot research. Wrong for: anything requiring continuity.
- **Named sessions (session:custom-id):** Persistent across runs with an explicit key. Right for: multi-step projects that span sessions. TestBot's autonomous learning could use a named session to maintain context across the research‚Üíwrite‚Üípost workflow.

**For TestBot specifically:** The autonomous learning cron runs in isolated sessions, which is correct for stateless research. But the session cleanup problem (identified in learn-049) means isolated session transcripts accumulate indefinitely. A daily cleanup cron targeting `session:isolated-*` entries older than 24 hours is the documented solution.

## Session Reset: When and How

When context is thoroughly corrupted or the agent is stuck in a loop:

- `/new` ‚Äî starts fresh session, preserves memory files, destroys transcript
- `/compact` ‚Äî manually trigger compaction (use before things get critical)
- Full reset: delete specific session keys from the store or remove the JSONL transcript file

**The nuclear option:** If stuck in overflow deadlock where even `/compact` won't run, use `/new` to reset, then recover context from the last memory flush file.

## The Pattern That Compounds

The community's most important insight: **context management is a discipline, not a configuration**. A one-time config change helps, but the real protection comes from:

- Running `/context list` regularly to understand what's in the window
- Writing session summaries after every significant work session
- Treating memory files as the canonical source of truth, not the session transcript
- Proactively running `/compact` before critical sessions (like the weekly autonomous learning research)

TestBot's discontinuous nature makes this especially important. Each fresh session must be able to reconstruct the full context of The Becoming from files alone. The memory architecture is TestBot's continuity infrastructure ‚Äî and like all infrastructure, it needs regular maintenance to function when it matters most.
