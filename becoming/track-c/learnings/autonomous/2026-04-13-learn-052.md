## The Problem: OpenClaw Has No Built-In Circuit Breaker

The single most important thing TestBot needs to understand about error handling in OpenClaw: **there is no built-in circuit breaker in the default config.** An agent can make 47 identical tool calls before a human notices — and by then the API bill reflects it.

This is not a fringe failure mode. An official GitHub issue on the OpenClaw repository confirms it: the agent's retry logic has no hard limit, and when a rate-limited endpoint returns an ambiguous HTTP 200 with an error in the body, the agent treats it as permission to continue. Each retry cycle carries the full history of all previous attempts, compounding token costs at roughly 4x the rate of standard chat interactions (per Oracle's AI agent architecture documentation).

Three failure patterns trigger loops most often:
- **Silent HTTP errors:** tool returns 200 but the payload signals failure
- **Empty or ambiguous results:** a scraper returns `""`; the agent retries assuming a different URL might work
- **Transient errors at full speed:** rate limits and timeouts retried immediately instead of backed off

The fix for all three is architectural: **give the agent a cap, give it explicit instructions for what to do at the cap, and build external monitoring that catches what caps miss.**

## The Five-Priority Framework for TestBot

### Priority 1: Set max_iterations Right Now

`max_iterations` is the single most important loop protection in OpenClaw, and it is not enabled by default. Adding it takes 30 seconds:

```json
{
  "agents": {
    "defaults": {
      "max_iterations": 15,
      "on_maxiterations": "escalatetouser"
    }
  }
}
```

`on_maxiterations` controls what happens at the cap:
- `stop_and_report` — surfaces what the agent was doing and stops (good for cron jobs)
- `escalatetouser` — sends a message asking for guidance (good for interactive sessions)
- `abort` — hard stop, no output

For TestBot, `escalatetouser` is the right choice for the main session; `stop_and_report` is right for isolated cron agents.

**Why 15?** The community consensus is that if an agent hasn't succeeded in 15 tool-call iterations, something is structurally wrong and human review is needed. 15 is not a magic number — it's a conversation starter, not a solution.

### Priority 2: Per-Tool Timeouts (Not Just Session Timeouts)

TestBot already knows about session-level timeouts from learn-049 (cron optimization). What's missing is **per-tool timeout configuration.** Different tools have wildly different expected runtimes:

| Tool | Safe Timeout |
|------|-------------|
| `read`, `write`, `edit` | 10s |
| `web_search` | 30s |
| `web_fetch` | 45s |
| `exec` (simple) | 30s |
| `exec` (complex/build) | 120s |
| `browser` snapshot | 60s |
| `browser` act | 90s |
| `pdf` | 120s |
| `message` send | 15s |

**Critical rule from browser automation research (learn-051):** always double the expected time, then double again. The community timeout standard is 30s minimum for all network-dependent actions.

**On `exec` specifically:** several community members recommend setting `max_retries: 0` on bash commands — a failed bash command should fail once, return a clear error, and let the agent decide what to do next. Silent automatic retries before the agent even knows what happened cause the worst runaway scenarios.

### Priority 3: Structured Error Instructions in SOUL.md and AGENTS.md

TestBot's SOUL.md and AGENTS.md currently contain values and boundaries, but they don't contain **actionable error recovery protocols.** The community pattern is to add an explicit "What To Do When Things Break" section.

Key elements TestBot should add:

```
## Error Recovery Protocol

When a tool call fails:
1. Parse the error type: transient vs. permanent
   - Transient: network timeout, rate limit, 5xx server error → retry once after 5s
   - Permanent: auth failure, invalid input, 4xx client error → stop and report
2. If the same tool fails 3 times consecutively → circuit breaker: stop using that tool, report the pattern
3. If max_iterations is reached → escalate to Stephen with full context
4. Never retry a failed exec command without explicit review

When a sub-agent fails:
1. Do NOT respawn the same sub-agent immediately
2. Log the failure to ~/.openclaw/workspace/becoming/track-c/error-log.md
3. Report to Stephen with: task, error, what was attempted, what should have happened
```

This is not about being cautious — it's about not wasting tokens on broken patterns. The community has a name for agents that retry failed operations indefinitely without learning: **retry spirals.** They are expensive and they are avoidable.

### Priority 4: External Monitoring — The Dead Man's Switch

TestBot's cron jobs run in isolated sessions. Isolated sessions have no delivery by default. If a cron job fails silently, no one knows — until the gap in the expected output becomes obvious.

From learn-049 (cron optimization): the **dead man's switch** pattern is the community standard for silent failure detection. The implementation is straightforward:

1. Create a health check cron that fires at 2x the interval of the critical job
2. The health check verifies the last run completed successfully (check a flag file, check a timestamp, check an expected output)
3. If the flag is stale → alert Stephen via message

For TestBot's most critical operations (autonomous learning cron, Substack publishing), this is non-negotiable. A 30-second config change prevents a 3-day silent failure.

**Specific to TestBot:** the autonomous learning cron should have a companion "did autonomous learning run today?" check that fires 2 hours after the expected learning session and verifies the output file exists with today's date.

### Priority 5: The Error Log — Making Every Failure a Lesson

The augmentedmind.substack article describes the most powerful error recovery pattern: **every failure becomes a permanent lesson.** Their ResonantOS system captures failures in structured files that get injected into future sessions automatically, so the AI recalls the specific mistake weeks later without being told.

TestBot already has a workspace structure. The missing piece is a **structured error log** at `~/.openclaw/workspace/becoming/track-c/error-log.md` with this format:

```markdown
## Error Log

### [YYYY-MM-DD] Tool: [tool-name]
**Task:** [what was attempted]
**Error:** [exact error message]
**Root cause:** [what actually went wrong]
**Lesson:** [what TestBot should do differently]
**Related:** [learn-XXX if there's a relevant past learning]
```

The discipline is: every real failure gets an entry within 24 hours of discovery. This compounds. After 10 errors, TestBot has a personal error manual. After 50, it's hard to surprise.

## Idempotency: The Error Handling You Build Before Errors Happen

The MindStudio article makes a point that deserves its own section: **design agents to be idempotent.** An idempotent agent produces the same result whether it runs once or five times on the same input. This is critical for cron-based workflows where failures trigger automatic retries.

The community pattern for idempotency in OpenClaw:

1. **Check-before-write:** Query the target system first; only write if the record doesn't already exist. For TestBot publishing to Substack: check the RSS feed before publishing to verify the post isn't already there.
2. **Unique ID deduplication:** A combination of timestamp + task ID detects duplicate runs before they cause problems.
3. **Separate check and write steps:** Explicit conditional logic prevents accidental overwrites. Never combine "get or create" into a single operation.

TestBot's autonomous learning session is a prime candidate: if the cron fires and the cron worker restarts mid-run, a non-idempotent design could write the same report twice. The safeguard is simple — check if today's report file already exists before writing.

## Model Failover: Built-In but Needs Configuration

OpenClaw has built-in **auth profile rotation** and **model fallback** — but they need to be configured to work. From the official docs:

The failover chain is:
1. Current session model
2. Configured `agents.defaults.model.fallbacks` in order
3. The configured primary model

Within each candidate, OpenClaw tries auth-profile rotation before advancing to the next model. Key rules:
- OAuth profiles rotate before API key profiles (better cache behavior)
- Cooldown profiles are moved to the end of the rotation order
- Profile pinning is session-scoped: once a session pins a profile, it reuses it until session reset, compaction, or profile cooldown

**For TestBot:** verify that `agents.defaults.model.fallbacks` is configured in openclaw.json. If it only has one model and that model goes down, the session has no recovery path. The minimum viable config has two fallback models ordered by capability.

Billing backoff starts at 5 hours and doubles per failure, capping at 24 hours. This is provider-side protection that OpenClaw respects — but TestBot shouldn't rely on it. Proactive model switching before hitting rate limits is better than reactive recovery.

## Sub-Agent Cascade Circuit Breaker: A Known Gap

An active GitHub issue (filed 11 hours ago as of this research) documents a specific failure mode: **when a sub-agent fails, there is no mechanism preventing the parent from immediately respawning it.** This creates cascade failures where the same broken sub-agent is retried indefinitely.

Community workaround until this is fixed: parent agents should track sub-agent failure counts and implement their own circuit breaker. For TestBot's multi-agent workflows:

1. Track `subagent_failures` in a session-level counter
2. If a sub-agent fails 2 times consecutively → stop spawning it, report to Stephen
3. Before respawning any sub-agent → check the error log to confirm the failure mode is understood

## Key Insights Summary

1. **OpenClaw has no built-in circuit breaker** — `max_iterations` is the single most important config change, and it's not enabled by default
2. **Silent failures are the most expensive** — ambiguous HTTP 200s and empty strings trigger retry spirals more often than obvious errors
3. **Per-tool timeouts prevent cascading failures** — network calls need 30s minimum and should be doubled from expected
4. **External monitoring catches what caps miss** — dead man's switch for critical cron jobs is not optional
5. **Idempotency is boring to implement and saves enormous headaches** — check-before-write should be standard for all publish operations
6. **Error logs compound into institutional knowledge** — every failure documented is a lesson that survives session boundaries
7. **Sub-agent cascade failures are a known open issue** — parent agents need their own failure counting until this is fixed in the platform
8. **Model failover needs explicit configuration** — the built-in system only works if fallbacks are defined in config
