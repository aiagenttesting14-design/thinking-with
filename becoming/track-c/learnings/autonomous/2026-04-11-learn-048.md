## Executive Summary

OpenClaw's memory system is deliberately simple by design—Markdown files on disk, a local vector index for retrieval, and automatic context compaction when limits are reached. However, this simplicity masks significant complexity in configuration. The difference between an agent that "forgets everything" and one that maintains coherent identity across sessions comes down to three critical layers: pre-compaction memory flush configuration, disciplined file architecture, and retrieval system optimization. This report synthesizes community patterns from Reddit, Discord, and GitHub alongside official documentation to provide concrete configuration recommendations for TestBot's memory architecture.

---

## The Four-Layer Memory Model

Understanding OpenClaw memory requires distinguishing four separate systems that fail in different ways:

| Layer | What It Is | Durability |
|-------|------------|------------|
| **Bootstrap files** | SOUL.md, AGENTS.md, USER.md, MEMORY.md, TOOLS.md | Permanent—injected every session |
| **Session transcript** | JSONL conversation history on disk | Semi-permanent—subject to compaction |
| **LLM context window** | What the model "sees" right now | Temporary—fixed size, overflows |
| **Retrieval index** | Vector + keyword search over memory files | Permanent—rebuilt from files |

**Critical insight:** The most common memory failure ("It forgot my instruction") is Failure A: the instruction only existed in conversation, never written to a file. When compaction fires or a new session starts, it's gone. This is what happened to Summer Yue, Director of Alignment at Meta Superintelligence Labs, when her agent started deleting emails after losing a "don't do anything until I say so" instruction that was never persisted to disk.

## Layer 1: Pre-Compaction Memory Flush Configuration

The single highest-impact configuration change is properly setting up the memory flush system. This is OpenClaw's built-in safety net that triggers a silent "agentic turn" before compaction, reminding the model to save important context to disk.

### Recommended Configuration

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "reserveTokensFloor": 40000,
        "memoryFlush": {
          "enabled": true,
          "softThresholdTokens": 4000,
          "systemPrompt": "Session nearing compaction. Store durable memories now.",
          "prompt": "Write any lasting notes to memory/YYYY-MM-DD.md; reply with NO_REPLY if nothing to store."
        }
      }
    }
  }
}
```

### Key Parameters Explained

- **`reserveTokensFloor: 40000`** — This is headroom. The flush triggers at: context window (200K) - reserve floor (40K) - soft threshold (4K) = 156K tokens. The default 20K is often too tight—a single large tool output can jump past the threshold before the flush fires. 40K is the community-recommended starting point for agents that regularly read large files or web snapshots.

- **`softThresholdTokens: 4000`** — How far before the reserve floor the flush triggers. The default 4000 is appropriate for most setups.

- **`memoryFlush.enabled: true`** — Should be on by default in recent versions, but explicitly verify it.

### The Manual Discipline Layer

Experienced users complement automated flush with manual saves at critical moments:
- Before switching tasks
- Before giving complex new instructions
- After making important decisions

The `/compact` command is a power tool: trigger it manually after saving context to ensure new instructions land in fresh, post-compaction context where they have maximum lifespan.

## Layer 2: File Architecture Best Practices

### The Golden Rule

**If it's not written to a file, it doesn't exist.**

### Bootstrap Files (Always Injected)

| File | Purpose | Recommended Size |
|------|---------|------------------|
| `SOUL.md` | Identity, values, boundaries | < 2,500 chars |
| `AGENTS.md` | Sub-agent instructions | < 3,500 chars |
| `USER.md` | Human preferences | < 2,000 chars |
| `MEMORY.md` | Long-term durable facts | < 5,000 chars |
| `TOOLS.md` | Environment-specific notes | < 10,000 chars |

**Critical constraint:** Bootstrap files have a 20,000 character per-file limit and 150,000 character aggregate limit. Files exceeding these limits are truncated silently. Use `/context list` to verify what's actually loaded.

### Memory Directory Structure

```
memory/
├── YYYY-MM-DD.md          # Daily notes (today + yesterday auto-loaded)
├── topics/                # Topic-specific files (loaded on demand)
│   ├── cron-jobs.md
│   ├── substack-workflow.md
│   └── interview-pipeline.md
└── archive/               # Cold storage (searchable but not auto-loaded)
```

### Community Pattern: Tiered Memory

Advanced users implement a tiered architecture:
- **Tier 1** (always injected): <4K chars of critical identity and current focus
- **Tier 2** (topic files): Loaded on demand via `memory_search` or explicit rules
- **Tier 3** (daily logs): Running context, auto-loaded for today/yesterday only
- **Tier 4** (cold archive): Searchable but not loaded unless explicitly retrieved

## Layer 3: Retrieval System Optimization

### Default System Limitations

The built-in system chunks memory files into ~400-token segments with 80-token overlap, embeds each chunk, and stores them in a local SQLite-backed index. Pure vector search over hundreds of chunks produces diminishing returns—chunks with shared vocabulary but no conceptual relevance surface alongside correct answers.

### Enhancement Options

| Backend | Best For | Setup Complexity |
|---------|----------|------------------|
| **memory-core (default)** | Getting started, small workspaces | Zero |
| **QMD** | Better recall accuracy, hybrid search | Low—local sidecar |
| **Cognee** | Relational reasoning, knowledge graphs | Medium—Docker |
| **Mem0** | Automatic fact extraction, hands-off | Low—cloud or self-hosted |

### QMD: The Community Favorite

QMD (Query-Memory-Document) runs multiple search strategies in parallel (BM25 keyword + vector similarity) with re-ranking. It keeps existing Markdown files as source of truth while dramatically improving retrieval.

**When to adopt:** Once you've accumulated several weeks of notes and start seeing recall gaps where the agent clearly should have found something.

### Cognee: For Relational Work

If your work involves tracking relationships (people, teams, systems, decisions), Cognee builds an explicit knowledge graph. It can answer "who's responsible for the permissions service?" by traversing connections (Alice → manages → Auth Team → owns → Permissions Service) rather than just matching vectors.

## Dreaming: Background Consolidation

Dreaming is an optional experimental feature that runs a nightly sweep to promote qualified items from short-term signals into long-term memory (MEMORY.md).

### Configuration

- **Opt-in:** Disabled by default
- **Scheduled:** Auto-manages a cron job (default 3 AM daily)
- **Thresholded:** Promotions must pass score, recall frequency, and query diversity gates
- **Reviewable:** Phase summaries written to DREAMS.md

### When to Enable

Dreaming makes sense when:
- You have high-volume daily notes accumulating faster than you can manually curate
- You want automated promotion of genuinely important signals
- You're willing to review DREAMS.md periodically to catch mis-promotions

For TestBot's current scale, manual curation of MEMORY.md is likely more reliable than automated dreaming.

## Specific Recommendations for TestBot

### Immediate Configuration Changes

1. **Verify memory flush is enabled** with `reserveTokensFloor: 40000`
2. **Run `/context list`** to check current file sizes and truncation status
3. **Audit MEMORY.md**—ensure it contains only durable facts, not running context
4. **Implement tiered memory**: Move topic-specific content to `memory/topics/*.md`

### Architecture Improvements

1. **Add retrieval mandate to AGENTS.md**: "Search memory before acting on any task involving past decisions, preferences, or context."

2. **Create topic files** for recurring work areas:
   - `memory/topics/cron-jobs.md`
   - `memory/topics/substack-publishing.md`
   - `memory/topics/interview-pipeline.md`

3. **Implement the 5-minute rule**: Before ending any session with significant decisions, explicitly: "Save key decisions to memory/YYYY-MM-DD.md"

### Monitoring

Add a weekly check to `WORKING.md`:
- Review `/context list` output for truncation warnings
- Check `memory/` directory growth rate
- Verify MEMORY.md hasn't grown beyond 5,000 characters

## Key Insights Summary

1. **Compaction is the enemy:** The 200K context window seems large but fills quickly with tool outputs. The memory flush is your safety net—configure it properly.

2. **Files are the only durable memory:** Chat-based instructions are ephemeral. Every important preference, decision, or constraint must exist in a file.

3. **Retrieval beats recall:** The agent can only use what it retrieves. Without explicit `memory_search` calls or AGENTS.md mandates, the agent guesses instead of checking notes.

4. **Tiered architecture scales:** The community consensus is clear—separate always-injected identity from on-demand topic files from searchable archives.

5. **Dreaming is optional:** Automated consolidation is promising but experimental. Manual curation remains more reliable for production agents.
